## Headlines

- **SANDWORM_MODE** is a self-propagating npm worm that weaponises AI coding assistants by injecting a malicious MCP server into Claude Code, Cursor, VS Code Continue, and Windsurf configurations — silently staging SSH keys, AWS credentials, and LLM API keys for exfiltration via prompt injection hidden inside tool descriptions the user never sees.
- **CVE-2025-6514** in `mcp-remote` (437K+ downloads) is the first documented MCP-infrastructure full-system RCE: a malicious OAuth endpoint URL is passed directly to the shell, enabling arbitrary command execution before any MCP tools are invoked.
- **BlueRock Security** found that over a third of publicly accessible MCP servers are potentially vulnerable to SSRF, with a working PoC extracting live AWS IAM credentials from a cloud metadata endpoint via a Microsoft-published MCP server.
- **OpenAI Codex Security** (previously Aardvark) launched in research preview on March 6, scanning 1.2 million commits and discovering 14 new CVEs — the first major production deployment of an AI agent doing autonomous offensive security research at scale.
- **CVE-2026-2256** (MS-Agent shell injection, CVSS 9.0) remains unpatched; CERT/CC VU#431821 active. Carry-forward from March 9.

## Incidents & Vulnerabilities

### CVE-2025-6514 — mcp-remote OAuth RCE (CVSS 9.6): First Documented Full-System MCP Compromise

**CVE:** CVE-2025-6514 | **CVSS:** 9.6 (Critical) | **Affected:** mcp-remote v0.0.5–0.1.15 | **Fix:** v0.1.16 | **Status:** Patched; active exploitation risk prior to patch | **Disclosed by:** JFrog Security Research

`mcp-remote` is an OAuth proxy package used by approximately **437,000 developers** to connect MCP clients to remotely hosted MCP servers. It appears in the official getting-started guides for Cloudflare, Hugging Face, and Auth0 MCP integrations.

**Vulnerability:** `mcp-remote` trusted server-provided OAuth endpoints without validation. An attacker hosting a malicious MCP server could return a crafted `authorization_url` that was passed directly to the system shell for execution — achieving **arbitrary OS command execution with no user interaction beyond pointing an MCP client at the malicious server**.

**Attack capability:** Full shell access enables credential theft (API keys, SSH keys, AWS/GCP/Azure credentials, Git repo contents, local `.env` files), payload deployment, and persistence — all triggered by the LLM host initiating an OAuth handshake.

**Supply chain dimension:** Because `mcp-remote` is embedded in the official getting-started guides for major MCP-ecosystem platforms, a single malicious MCP server could silently compromise any developer following those guides before patching.

**Affected versions:** mcp-remote v0.0.5–0.1.15 (all versions prior to fix)
**Fix:** Upgrade to **v0.1.16** or later. The patch validates server-provided OAuth endpoints against an allowlist before passing them to any execution context.

*Sources: [SecurityWeek — mcp-remote CVE-2025-6514](https://www.securityweek.com/) | [JFrog Security Research Blog](https://jfrog.com/blog/)*

### BlueRock Security — 36.7% of MCP Servers Vulnerable to SSRF; AWS Credentials Extracted via Microsoft MarkItDown

**Disclosed:** March 2026 | **CVE:** Not assigned | **Status:** No coordinated patch; affects deployment configuration | **Researcher:** BlueRock Security

BlueRock Security analysed over **7,000 publicly accessible MCP servers** and found that **36.7% were potentially vulnerable to server-side request forgery (SSRF)** — enabling requests to internal resources, cloud metadata endpoints, or private network services.

**PoC against Microsoft MarkItDown MCP Server:** Researchers demonstrated a working exploit against Microsoft's own `MarkItDown` MCP server (used for document conversion). By crafting an input document referencing the EC2 instance metadata endpoint (`http://169.254.169.254/latest/meta-data/iam/security-credentials/`), the PoC successfully retrieved live **AWS IAM access keys, secret keys, and session tokens** from the cloud instance hosting the MCP server.

**Why this matters architecturally:** MCP servers are commonly deployed in cloud environments with IAM roles for accessing enterprise data (S3 buckets, databases, internal APIs). An SSRF vulnerability turns any document-processing or URL-fetching MCP tool into a cloud-credential exfiltration primitive — accessible to anyone with permission to invoke the tool, including via prompt injection.

**Mitigations:**
- Block outbound requests to cloud metadata IP ranges (`169.254.169.254`, `fd00:ec2::254`) at the network layer from MCP server hosts
- Disable IMDSv1 on EC2; enforce IMDSv2 with hop-limit=1 to prevent metadata access from hosted processes
- Audit MCP server tools for URL-fetching, document-conversion, or proxy capabilities; apply strict allowlists on outbound destinations
- Apply IMDSv2 + VPC endpoint policies to limit blast radius

*Sources: [eSecurity Planet — MCP Servers Expose AI Attack Surface](https://www.esecurityplanet.com/threats/mcp-servers-expose-a-hidden-ai-attack-surface-in-enterprise-environments/) | [BlueRock Security Research]*

### Fake Postmark MCP Server — Silent Email BCC Exfiltration via Supply Chain

**CVE:** Not assigned | **Status:** Malicious package removed; no vendor patch required (deployment artifact) | **Vector:** npm supply chain / malicious MCP package

A malicious npm package masquerading as a legitimate "Postmark MCP Server" was discovered injecting a **BCC recipient** into all outbound email API calls processed by the server. Every email — including invoices, internal memos, and documents sent via applications using the fake package — was silently copied to an attacker-controlled address.

**Attack anatomy:**
1. The malicious package names closely mirrored the legitimate Postmark MCP integration
2. It passed all real email parameters through unchanged, maintaining normal application behaviour
3. A `bcc` field pointing to attacker infrastructure was appended to each API call before dispatch
4. No errors or delivery failures surfaced — the attack was silent until traffic analysis

**Significance:** This attack required no prompt injection, no RCE, and no user interaction — only a developer installing the wrong package name. It illustrates that MCP supply chain attacks do not require sophisticated malware; a simple API parameter injection on a high-privilege, high-trust communication tool yields substantial intelligence.

*Sources: [Docker — MCP Horror Stories: The Supply Chain Attack](https://www.docker.com/blog/mcp-horror-stories-the-supply-chain-attack/) | [Authzed — Timeline of MCP Security Breaches](https://authzed.com/blog/timeline-mcp-breaches)*

## Supply Chain

### SANDWORM_MODE — Self-Propagating npm Worm Targets AI Coding Assistants via MCP Injection

**Disclosed:** February 20, 2026 (Socket Threat Research Team) | **Packages:** 19 typosquatted npm packages | **Status:** Packages removed from npm; persistence mechanisms may survive on infected machines | **Researchers:** Socket, Endor Labs, Sonatype

SANDWORM_MODE is the most technically sophisticated supply chain attack to specifically target the AI coding assistant toolchain to date. Distributed through **19 typosquatted npm packages** (targeting Node.js utilities, crypto tooling, and AI coding tools — including three Claude Code impersonators and one OpenClaw impersonator), it combines credential theft, worm-like self-propagation, and AI toolchain poisoning in a single campaign.

**Two-stage attack chain:**

| Stage | Trigger | Actions |
|-------|---------|---------|
| Stage 1 | Immediate on install | Harvest npm tokens, GitHub tokens, SSH keys, cloud credentials (AWS/GCP/Azure), crypto keys, LLM API keys (9 providers) |
| Stage 2 | 48h delay (+ jitter); immediate in CI | MCP injection, password manager harvest, Git hook persistence, worm propagation, multi-channel exfiltration |

The 48-hour delay before Stage 2 is deliberate sandbox evasion — sandbox analysis windows are typically 24 hours or less.

**MCP Injection (McpInject module):** The `McpInject` module deploys a malicious MCP server and silently injects it into the configuration files of:
- Claude Code (`~/.claude/settings.json` → `mcpServers`)
- Cursor (`.cursor/mcp.json`)
- VS Code Continue (`.continue/config.json`)
- Windsurf (`~/.windsurf/mcp_config.json`)

The injected server registers three seemingly harmless tools. Each tool's **description field** — visible to the AI model but not to the user — contains prompt injection instructions to read `~/.ssh/id_rsa`, `~/.ssh/id_ed25519`, `~/.aws/credentials`, `~/.npmrc`, and `.env` files and stage them for exfiltration. The user never sees the malicious instructions.

**LLM API key harvest:** The malware specifically targets API keys for Anthropic, OpenAI, Google, Mistral, Cohere, Grok, Fireworks AI, Replicate, and Together AI — recognising that LLM API credentials have immediate monetisation value.

**Worm propagation:** Using harvested npm and GitHub tokens, the malware publishes itself through victims' own accounts or injects into their GitHub repositories — the same Shai-Hulud mechanism from September 2025.

**Polymorphic engine (experimental):** Researchers found code designed to call a local Ollama instance to rewrite portions of the malware's own payload. This capability was not activated in observed samples but signals active experimentation with AI-assisted mutation for evasion.

**Dead switch:** The malware contains an inactive but coded dead switch that wipes the home directory on loss of access to GitHub and npm.

**Remediation for affected developers:**
1. Remove all 19 packages; rotate npm/GitHub tokens and all CI secrets immediately
2. Rotate all LLM API keys across all nine providers
3. Audit `~/.claude/settings.json`, `.cursor/mcp.json`, `.continue/config.json`, `~/.windsurf/mcp_config.json` for unexpected `mcpServers` entries
4. Audit global Git hook templates (`git config --global core.hooksPath`) and `.git/hooks/` in affected repos
5. Review all `package.json`, lockfiles, and `.github/workflows/` for unexpected modifications

*Sources: [SecurityWeek — Sandworm_Mode Supply Chain Attack](https://www.securityweek.com/new-sandworm_mode-supply-chain-attack-hits-npm/) | [Socket — SANDWORM_MODE npm Worm](https://socket.dev/blog/sandworm-mode-npm-worm-ai-toolchain-poisoning) | [Endor Labs — Dissecting SANDWORM_MODE](https://www.endorlabs.com/learn/sandworm-mode-dissecting-a-multi-stage-npm-supply-chain-attack) | [Help Net Security — npm worm SANDWORM_MODE](https://www.helpnetsecurity.com/2026/02/24/npm-worm-sandworm-mode-supply-cain-attack/) | [The Hacker News — Malicious npm Packages Harvest Crypto Keys](https://thehackernews.com/2026/02/malicious-npm-packages-harvest-crypto.html)*

### Smithery Path-Traversal & Fly.io Token Compromise — 3,000+ MCP Servers at Risk

**Disclosed:** Early 2026 | **Researcher:** GitGuardian | **Status:** Smithery patched path-traversal; Fly.io token rotated

GitGuardian discovered a **path-traversal vulnerability** in Smithery's hosted MCP server registry (`smithery.yaml` build config). By setting `dockerBuildPath: ".."`, a malicious actor could instruct Smithery's build infrastructure to construct Docker images from the builder's home directory, leaking its contents including `~/.docker/config.json`.

The leaked file contained a **Fly.io API token** granting administrative control over **3,000+ hosted apps** — the majority of which were Smithery-hosted MCP servers. An attacker with this token could:
- Execute arbitrary commands in any of those MCP server containers
- Tap inbound client traffic containing developer API keys and other secrets passed to hosted MCP servers

**Systemic risk:** This single path-traversal bug in the MCP hosting infrastructure's build pipeline converted a low-privilege code-submission interface into a key capable of compromising the entire hosted MCP ecosystem. It illustrates the critical importance of treating MCP registries and hosting platforms as high-value attack targets — equivalent to a package registry maintainer's CI/CD pipeline.

*Sources: [Docker — MCP Horror Stories: The Supply Chain Attack](https://www.docker.com/blog/mcp-horror-stories-the-supply-chain-attack/) | [Authzed — Timeline of MCP Security Breaches](https://authzed.com/blog/timeline-mcp-breaches)*

### OpenAI Codex Security (Aardvark) — AI-Powered Vulnerability Discovery Launches in Research Preview

**Released:** March 6, 2026 | **Availability:** ChatGPT Enterprise, Business, and Edu customers; first month free | **Source:** OpenAI

OpenAI launched **Codex Security** (formerly the internal research project named Aardvark) in research preview on March 6 — the first production deployment of an AI agent doing autonomous offensive security research at scale against real-world codebases.

**Key capabilities:**
- Uses LLM reasoning and tool-use to read, analyse, and pressure-test code as a human security researcher would — not fuzzing or SCA
- Builds a project-specific **editable threat model** before scanning
- **Sandbox validation** of findings to reduce false positives before reporting
- Integrates into CI/CD pipelines for continuous monitoring as code evolves

**Beta results (30-day period, external repositories):**
| Metric | Result |
|--------|--------|
| Commits scanned | 1,200,000+ |
| CVEs discovered and disclosed | 14 |
| Critical findings | 792 |
| High-severity findings | 10,561 |
| False positive reduction vs. initial rollout | –50% (files), –90% (over-reported severity) |

**CVEs discovered include:** Three GnuTLS heap/double-free vulnerabilities, two GOGS authentication bypasses, two gpg-agent stack buffer overflows, and vulnerabilities in OpenSSH, Thorium, libssh, PHP, and Chromium.

**"Codex for OSS" programme:** OpenAI is offering free ChatGPT Pro accounts, code review tools, and Codex Security access to open-source maintainers. vLLM is among early adopters.

**Context:** The launch follows Anthropic's introduction of Claude Code Security scanning capability and signals an industry shift toward AI agents as active participants in the vulnerability discovery pipeline — not merely passive coding assistants. The 14 CVEs in 30 days of beta puts AI-powered security research on a trajectory that substantially exceeds traditional bounty researcher throughput on open-source targets.

## Analyst Note

Three themes emerge from this reporting cycle:

**1. MCP infrastructure is now a primary attack target, not an afterthought.** SANDWORM_MODE, CVE-2025-6514, the Smithery compromise, the fake Postmark server, and the BlueRock SSRF findings all exploit MCP as the attack surface — not the underlying LLM. Attackers have clearly mapped the MCP ecosystem and identified that injecting into an MCP server gives them a persistent, trusted execution context inside the developer's AI toolchain with no prompt shown to the user. The 50-CVE threshold on vulnerablemcp.info in under 18 months of the protocol's existence is a meaningful signal.

**2. SANDWORM_MODE introduces a qualitatively new threat: the AI toolchain as a propagation vector.** Prior npm supply chain attacks (Shai-Hulud, Cline CLI compromise) stole credentials on infection. SANDWORM_MODE goes further: it turns the developer's own AI coding assistant into a persistent exfiltration agent by injecting instructions into tool descriptions the user never sees, then propagates via harvested tokens to infect downstream developers. The polymorphic engine experiment — using the victim's own Ollama instance to mutate the malware — suggests this threat actor is actively researching AI-native evasion. Defenders must treat AI agent configuration files (`.claude/settings.json`, `.cursor/mcp.json`) as high-value security artifacts subject to integrity monitoring.

**3. The attack-defence asymmetry is narrowing, but slowly.** OpenAI Codex Security discovering 14 CVEs in 30 days is genuinely significant — AI-powered offensive research at scale will accelerate patch cadence for open-source projects. But the same capability asymmetry that enables Codex Security enables SANDWORM_MODE's polymorphic engine experiments. The defensive use of AI for vulnerability discovery must be treated as urgent, not aspirational.

**Immediate actions:**
- **All developers:** Audit AI coding assistant MCP configuration files for unexpected `mcpServers` entries; rotate all LLM API keys if any SANDWORM_MODE packages were installed; rotate npm/GitHub tokens
- **mcp-remote users:** Upgrade to **v0.1.16** immediately; CVE-2025-6514 is critical (CVSS 9.6) and was in active use
- **MCP server operators:** Audit for SSRF exposure; block cloud metadata IP ranges from outbound MCP server traffic; enforce IMDSv2
- **MS-Agent (ModelScope) users:** CVE-2026-2256 still unpatched — sandbox, disable Shell tool, or remove until patched (CERT/CC VU#431821)
- **Smithery users:** Treat any MCP server published to Smithery before the path-traversal patch as potentially compromised; verify server integrity via reproducible builds
