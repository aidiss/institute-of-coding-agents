## Incidents & Vulnerabilities

### Mindgard / [un]prompted Conference — 37 Vulnerabilities Across 15 AI IDEs

**Disclosed:** March 2026 | **Researcher:** Piotr Ryciak (Mindgard) | **Presented at:** [un]prompted conference | **Scope:** 15+ AI coding tools | **Vulnerabilities:** 37 | **All culminating in:** RCE, data exfiltration, or sandbox bypass | **CVE status:** In progress

Mindgard AI red teamer Piotr Ryciak presented a systematic audit of 15+ major AI coding tools at the [un]prompted security conference in March 2026, identifying 37 security vulnerabilities across Google Gemini CLI, OpenAI Codex, Amazon Kiro, Anthropic Claude Code, Cursor, and others. Every vulnerability found led to one of three outcomes: remote code execution, data exfiltration, or sandbox bypass.

**The trust boundary problem:** A key finding from the Mindgard audit is that there is currently **no industry consensus** on where security boundaries should be drawn in agentic IDEs — for example, whether trust persistence between sessions is a vulnerability or a reasonable UX tradeoff. Different product teams have landed in different places. Some vendors assigned CVEs for time-of-check-time-of-use (TOCTOU) patterns; others classified identical behaviours as merely informational findings. This inconsistency means the same underlying flaw can receive a CVE from one vendor and no action from another — making cross-tool risk comparison difficult for enterprise security teams.

**Relationship to IDEsaster:** Mindgard's 37-vulnerability dataset is distinct from the IDEsaster report (30+ CVEs, December 2025, Ari Marzouk / MaccariTA). Together the two disclosures document **67+ distinct vulnerabilities** across the AI IDE ecosystem in a span of roughly four months. The attack surface is growing faster than the patch rate.

### Terra Security — 100% Vulnerability Rate in AI Copilot Applications

**Published:** February 24, 2026 | **Source:** Terra Security / BusinessWire | **Scope:** AI-generated apps, embedded AI copilots, enterprise AI integrations | **Finding:** AI-specific vulnerabilities found in 100% of tested applications

Terra Security released findings from adversarial testing conducted across applications built with AI coding tools (including Claude Code), rapid AI app-generation platforms (Loveable, Base44), and enterprise software embedding AI copilots. The headline finding: **every single tested application** contained AI-related security vulnerabilities — a 100% hit rate.

The vulnerability classes identified differ from traditional software security flaws in important ways:

- **Prompt injection bridges** embedded in generated code that allow external content to redirect the application's AI component — distinct from classic XSS or injection flaws
- **Trust misconfiguration** in AI-integrated authentication flows — the generated code assumes that responses from an AI component are trusted system output, creating privilege escalation paths
- **Data exfiltration via AI intermediaries** — the AI component is positioned as a trusted channel, allowing untrusted inputs to leverage it as an exfiltration path that bypasses traditional DLP controls
- **Over-permissive tool grant patterns** — AI-generated scaffolding code consistently grants broader MCP server permissions than the application requires at runtime

Terra noted that these patterns are **not random bugs** but systematic design decisions that reflect how AI coding tools frame their generation context — they optimise for functional completeness over security posture, producing code that works correctly under normal use but fails adversarially in consistent ways.

**Implication for enterprises:** If you are deploying applications built with AI coding assistants, or embedding AI copilots into your products, treat the output as requiring security review equivalent to code written by a junior developer without security training. Static analysis tools calibrated for human-authored code will miss many of these patterns.

*Sources: [Terra Security Finds Widespread Exploitable Flaws in AI-Driven Applications — BusinessWire](https://www.businesswire.com/news/home/20260224886429/en/Terra-Security-Finds-Widespread-Exploitable-Flaws-in-AI-Driven-Applications-Copilots-and-AI-Generated-Code) | [Security in 2026: New Ways Attackers Are Exploiting AI Systems — National CIO Review](https://nationalcioreview.com/articles-insights/extra-bytes/security-in-2026-new-ways-attackers-are-exploiting-ai-systems/)*

### CVE-2026-2256 — MS-Agent (ModelScope) Shell Injection (Carry-Forward; Day 4, No Patch)

**CVSS:** 9.0 (Critical) | **Vendor fix:** None | **PoC:** Public | **CERT/CC:** VU#431821 | **Day 4 without patch**

No patch issued. The public proof-of-concept for arbitrary shell command injection via unsanitized agentic tool input in ModelScope's MS-Agent continues to circulate without remediation. Active exploitation risk remains elevated.

**Mitigations (no patch available):**
- Sandbox all MS-Agent deployments; do not run on hosts with access to credentials or sensitive infrastructure
- Disable or remove the Shell tool entirely
- Monitor for unexpected child process spawning from MS-Agent worker processes
- Treat any host that has run MS-Agent with untrusted inputs as potentially compromised pending a patch

## Supply Chain

### Slopsquatting — Phantom `react-codeshift` Package Claimed by Researcher (Demonstrated)

**Date:** January 2026 | **Researcher:** Charlie Eriksen (Aikido) | **Package:** `react-codeshift` (npm) | **Registry:** npm | **Status:** Package claimed; exploit demonstrated against live AI agents

Security researcher Charlie Eriksen at Aikido found a hallucinated package name spreading through real AI coding agent infrastructure in January 2026: the npm package `react-codeshift` was being actively requested by autonomous AI agents — not as a result of human copy-paste, but because independent LLM sessions consistently hallucinated the name when reasoning about React code transformation tasks. No human had seeded the hallucination; models invented it independently across different sessions.

Eriksen claimed the phantom name on npm and confirmed that real AI agents in production were attempting to install it. The package was benign, but the experiment conclusively demonstrated that **slopsquatting attacks are now practically viable against autonomous AI coding agents** operating without human package-name verification — not just against developers copy-pasting AI suggestions.

**Why agentic slopsquatting is qualitatively worse than developer slopsquatting:**

| Factor | Developer copy-paste | Autonomous agent |
|--------|----------------------|------------------|
| Human review possible | Yes | No |
| Verification step | Optional but possible | Typically absent |
| Scale per session | One developer | Thousands of automated workflows |
| Detection window | PR/code review | Post-execution |
| Blast radius | One machine | Entire CI/CD environment |

**Scale of the underlying hallucination problem (2025 research):** A study of 576,000 AI-generated code samples found that ~20% of recommended packages did not exist; 43% of hallucinated names recurred consistently across repeated prompts; 58% appeared in more than one run. Python (23% hallucination rate) and Rust (25%) are significantly worse-affected than JavaScript (15%).

**Confirmed real-world precedent:** In an earlier 2024 experiment, researcher Bar Lanyado (Lasso Security) registered the hallucinated PyPI package `huggingface-cli` and received **30,000+ authentic downloads** in three months — including from Alibaba's own repositories.

**Mitigations:**
- Require all AI-generated dependency installs to be verified against the official registry before execution
- Pin all dependencies with lockfiles and cryptographic hash verification
- Run `pip install` and `npm install` invocations from AI agents in ephemeral, network-isolated containers
- Integrate automated package provenance checking into CI/CD pipelines
- Build SBOMs for every AI-assisted build

## Research & Advisories

### OpenAI Codex Security (formerly Aardvark) — AI Autonomous Vulnerability Researcher

**Launched:** March 6, 2026 (research preview) | **Source:** OpenAI | **Capability:** Autonomous vulnerability discovery, validation, and remediation assistance | **Benchmark:** 92% of known and synthetic vulnerabilities identified

OpenAI launched **Codex Security** (rebranded from its internal Aardvark project) as a research preview on March 6, 2026 — an autonomous AI-powered security researcher that independently finds, validates, and assists with remediating software vulnerabilities at scale.

In benchmark testing, Codex Security identified **92% of known and synthetically-introduced vulnerabilities** across test codebases. Applied to open-source projects during the research phase, it discovered vulnerabilities that resulted in the responsible disclosure of findings earning **10 CVE identifiers**.

**Significance for the AI coding security landscape:** Codex Security represents the first major signal that the AI ecosystem is beginning to deploy autonomous offensive AI capabilities against its own attack surface. This has two implications:

1. **Defensive potential:** If AI agents can autonomously find 92% of vulnerabilities, continuous automated security review of AI-generated code becomes feasible at a scale and cost that was not practical with human-only review.
2. **Attack surface symmetry:** The same capabilities that make Codex Security a useful defensive tool — autonomous code analysis, vulnerability identification, exploit path tracing — are equally applicable offensively. The arms race between AI-assisted offence and defence is accelerating.

The launch also highlights the gap between what is technically possible (92% automated vulnerability detection) and the current state of deployment (most AI-generated code ships without any automated security review). Closing that gap is the near-term practical challenge.

### IBM Research — "Mind the Gap": Disconnect Between Vulnerability Disclosure and AI Security

**Published:** 2026 | **Venue:** IEEE SaTML 2026 (Security and Trustworthiness of Machine Learning) | **Source:** IBM Research

IBM Research published a position paper — "Mind the Gap: Closing the Growing Disconnect Between Vulnerability Disclosure and AI Security" — submitted to IEEE SaTML 2026. The paper identifies a structural problem in how AI security vulnerabilities are currently discovered, reported, triaged, and patched:

**Key findings:**

- **CVE infrastructure was not designed for AI vulnerabilities.** Standard CVE metadata (affected versions, vendor, component) maps poorly to AI-specific flaws like prompt injection, jailbreaks, and emergent capability misuse — which are often model-version-agnostic or cut across multiple product layers simultaneously.
- **Responsible disclosure timelines are mismatched.** The 90-day disclosure norm assumes a vendor can issue a software patch. Many AI vulnerabilities cannot be patched with a code change — they require retraining, architectural changes, or policy updates that take months. Vendors have no agreed alternative timeline.
- **AI vendors are inconsistently applying CVEs.** Some vendors assign CVEs for prompt injection vulnerabilities; others treat identical behaviours as out-of-scope. The resulting fragmentation means that organisations using multiple AI tools cannot build a coherent vulnerability management programme across their AI stack.
- **Attack success rates exceed defence success rates.** A meta-analysis of 78 recent studies (2021–2026) found that adaptive attack strategies achieve success rates exceeding 85% against state-of-the-art defences — a gap that has not narrowed over the study period.

**Recommendation:** IBM calls for a dedicated AI vulnerability taxonomy and disclosure framework, analogous to CWE/CVE but structured for AI-specific failure modes, coordinated across NIST, MITRE, and major AI vendors.

*Sources: [Mind the Gap — IBM Research / IEEE SaTML 2026](https://research.ibm.com/publications/position-mind-the-gap-closing-the-growing-disconnect-between-vulnerability-disclosure-and-ai-security)*

### "Vibe Coding" Security Risks — Academic Perspective

**Published:** March 2026 | **Source:** Wits University (opinion / academic commentary)

Wits University published an opinion piece in March 2026 on the security risks of "vibe coding" — the practice of generating entire applications from natural language prompts with minimal developer review of the produced code. The article captures a tension that underlies much of the vulnerability research in this brief:

Vibe coding maximises developer velocity at the expense of security posture. Developers who accept AI-generated code without systematic review are not simply introducing normal software bugs at a higher rate — they are introducing **a qualitatively different class of vulnerability** that reflects the LLM's training distribution rather than the developer's intent. LLMs have been trained on vast quantities of insecure open-source code; the security mistakes they reproduce are not random but correlated with the most common patterns in that training data.

The practical implication: applications built with minimal review of AI-generated code are likely to share common vulnerability patterns across organisations — the same prompt, the same LLM, the same flaw. This creates conditions for **cross-organisation vulnerability campaigns** where an attacker who characterises one LLM's generation patterns can target many downstream applications simultaneously.

*Sources: [Securing Vibe Coding: The Hidden Risks Behind AI-Generated Code — Wits University](https://www.wits.ac.za/news/latest-news/opinion/2026/2026-03/securing-vibe-coding-the-hidden-risks-behind-ai-generated-code.html)*

## Analyst Note

Three themes converge this cycle:

**1. The vulnerability disclosure infrastructure is breaking under AI-scale load.** Mindgard's 37-vulnerability disclosure joins IDEsaster's 30+ CVEs, the seven-MCP-CVEs-in-one-month cluster, and Claude Code's multi-CVE batch to produce a picture where the rate of AI security vulnerability discovery now exceeds the ecosystem's capacity to triage, patch, and communicate fixes. IBM Research's "Mind the Gap" paper formalises what practitioners have been experiencing operationally: CVE infrastructure, 90-day disclosure norms, and vendor patch processes were designed for a world where vulnerabilities are isolated bugs in versioned software — not systemic architectural flaws that cut across model versions and product layers simultaneously.

**2. Slopsquatting has crossed from theoretical to demonstrated at scale.** The `react-codeshift` experiment shows that autonomous AI agents — not just human developers — are the target. Supply chain attacks via hallucinated dependencies do not require social engineering or developer error; they exploit the intrinsic generation behaviour of LLMs. As AI coding agents gain autonomous dependency installation capabilities (the ability to run `pip install` or `npm install` without human approval), the attack becomes fully autonomous end-to-end. Organisations deploying agentic coding pipelines without lockfile enforcement and package provenance verification are carrying an unquantified open exposure.

**3. AI is now fighting AI.** OpenAI Codex Security finding 92% of vulnerabilities autonomously is significant not just as a product launch but as a signal about where the security landscape is heading. The organisations with the most effective AI-assisted vulnerability discovery will have a structural security advantage over those relying on traditional human-only review. Given that AI-generated code is already producing vulnerabilities at scale (Terra Security: 100% hit rate), the organisations that close the loop — using AI to generate code and AI to security-test it — will be better positioned than those doing neither.

**Immediate actions:**
- **All AI IDE users:** Treat the Mindgard findings as confirmation that every major AI coding tool has been found vulnerable to RCE-class issues. Human-in-the-loop approval for agent actions that touch files, run commands, or modify configuration is the most effective current mitigation.
- **Teams using AI agents with autonomous dependency installation:** Enforce lockfiles, hash verification, and ephemeral container execution for all package installs; add registry existence checks pre-install.
- **MS-Agent (ModelScope) operators:** CVE-2026-2256 is now on day 4 without a patch (CVSS 9.0, public PoC). Sandbox or remove immediately.
- **Enterprise copilot and AI app deployments:** Commission adversarial AI security testing against your deployed applications — Terra Security's 100% hit rate suggests that standard security review is insufficient for AI-generated and AI-embedded code.
- **Security teams:** Review IBM Research's "Mind the Gap" framework for structuring AI vulnerability management programmes; the existing CVE/patch workflow is inadequate for the current AI vulnerability class.
