## Headlines

- **CVE-2026-0755** (gemini-mcp-tool, CVSS 9.8) remains unpatched after ZDI published it as a zero-day on January 9 following seven months of vendor non-response — network-reachable, no auth, no user interaction required; no fix available.
- **GitHub Copilot** accumulated two new high-severity CVEs this cycle: CVE-2026-21516 (command injection, CVSS 8.8, no auth required) and CVE-2026-29783 (CLI shell-expansion patterns enabling RCE via prompt-injected repo files); CLI fix available in v0.0.423+.
- **CVE-2026-21858** (n8n, CVSS 10.0): unauthenticated attackers can fully take over n8n agentic workflow platform deployments, execute arbitrary commands on the host, and extract enterprise data pipelines; active exploitation in the wild confirmed.
- **PleaseFix / PerplexedBrowser**: Zenity Labs disclosed zero-click agent compromise and credential theft paths in Perplexity Comet's agentic browser (March 4); Perplexity patched prior to public disclosure.
- **s1ngularity** (Nx Build System): first confirmed instance of npm supply chain malware turning AI coding CLI tools (Claude, Gemini, Amazon q) into active reconnaissance and exfiltration instruments.
- **Anthropic mcp-server-git**: three chained CVEs (CVE-2025-68143/68144/68145) disclosed January 20 by Cyata researcher Yarden Porat — path traversal + argument injection achieving RCE through prompt injection alone; patched.
- **CVE-2026-2256** (MS-Agent shell injection, CVSS 9.0) carries forward for the third consecutive day with no vendor patch. CERT/CC VU#431821 active.

## Incidents & Vulnerabilities

### CVE-2026-0755 — gemini-mcp-tool Command Injection Zero-Day (CVSS 9.8, Unpatched)

**CVE:** CVE-2026-0755 | **CVSS:** 9.8 (Critical) | **ZDI Advisory:** ZDI-26-021 | **Affected:** gemini-mcp-tool all versions | **Fix:** None | **Status:** Unpatched zero-day; public advisory since January 9, 2026 | **Disclosed by:** Trend Micro Zero Day Initiative

`gemini-mcp-tool` is an MCP integration package enabling Google Gemini models to invoke shell operations as part of agentic tool use. ZDI discovered CVE-2026-0755 on **July 25, 2025**, notified the vendor, set a disclosure deadline, received no response through November 2025, issued a final warning in December 2025, and published the advisory as a zero-day on **January 9, 2026**. The vendor has remained unresponsive.

**Technical root cause:** The tool's `execAsync` wrapper passes unsanitized user-controlled input directly to Node.js `child_process.exec`. Because `exec` invokes a shell to interpret the command string, standard shell metacharacters (`; | & $() `` \n`) allow full injection. An attacker can break out of the intended command context and execute arbitrary OS commands with the privileges of the service account running the MCP server.

**Exploit path:** The vulnerability is **network-reachable** (MCP clients connect over the network), requires **no authentication**, and requires **no user interaction**. Successful exploitation grants full shell access — enabling credential theft, payload deployment, persistence, and lateral movement. The LLM itself may refuse to generate obviously malicious tool calls, but this client-side filtering is not a security boundary: prompt injection bypasses it reliably.

**CWE:** CWE-78 (OS Command Injection)

**Mitigations (no patch available):**
- Replace `exec` / `execAsync` with `child_process.execFile` or `spawn` with argument arrays (eliminates shell interpolation)
- Run gemini-mcp-tool in a Docker/Podman container with restricted network egress and read-only filesystem mounts
- Apply least-privilege service account; do not run as root or a credential-bearing IAM role
- Disable the `execAsync` functionality entirely if not operationally required

### CVE-2026-21516 / CVE-2026-29783 — GitHub Copilot Command Injection Pair

**CVE-2026-21516 — GitHub Copilot Service Command Injection**

**CVE:** CVE-2026-21516 | **CVSS:** 8.8 (High) | **Affected:** GitHub Copilot service | **Fix:** Vendor patch applied server-side | **Status:** Patched | **Auth required:** None

An unauthenticated attacker able to reach the GitHub Copilot service over the network can trigger arbitrary command injection without user interaction. The flaw scores 8.8 because it requires no authentication and is exploitable over any network path that can reach the service endpoint. The attack surface includes developers whose Copilot installations are reachable through misconfigured network policies or through prompt injection that drives Copilot to invoke the vulnerable path against attacker-controlled content.

**CVE-2026-29783 — GitHub Copilot CLI Shell Expansion RCE**

**CVE:** CVE-2026-29783 | **Affected:** `@github/copilot` npm CLI package < v0.0.423 | **Fix:** Upgrade to v0.0.423 or above | **Status:** Patched | **Vector:** Prompt injection via repository files

The GitHub Copilot CLI's shell tool contains dangerous bash parameter expansion patterns that can be triggered through crafted inputs in repository files. When an attacker embeds payload-bearing content in files that Copilot's CLI agent reads as part of a task (README, code comments, configuration files), the expansion patterns enable arbitrary code execution on the developer's machine.

The attack chain is: attacker commits malicious file → developer asks Copilot CLI to work with that file → CLI processes expansion → arbitrary code executes. GitHub's response was to add a display warning on github.com for files containing hidden Unicode characters (following Pillar Security's Rules File Backdoor disclosure), but the CLI code fix requires explicit package upgrade.

**Action required:** Upgrade `@github/copilot` CLI to **v0.0.423 or above**.

### CVE-2026-21858 — n8n AI Workflow Platform RCE (CVSS 10.0)

**CVE:** CVE-2026-21858 | **CVSS:** 10.0 (Critical) | **Affected:** n8n self-hosted deployments | **Fix:** Patch available; upgrade immediately | **Status:** Actively exploited in the wild

n8n is a widely deployed open-source platform for building LLM-powered agentic workflows. Organisations use it to connect AI models to internal databases, APIs, and data pipelines in production. A critical unauthenticated RCE vulnerability allows an attacker who can reach a self-hosted n8n instance over the network to **completely take over the host**, execute arbitrary system commands, and extract all data handled by the platform's workflow pipelines.

**Why this matters for AI coding security:** n8n is commonly deployed as the orchestration layer for AI coding automation pipelines — running code generation agents, code review agents, CI/CD integration, and repository management automation. A compromised n8n host grants an attacker full control over every agentic workflow on that host, including those with repository write access, secret access, and CI/CD trigger permissions. The blast radius scales with the privilege level of the agents running on the platform.

**Immediate action:** Patch n8n immediately. If patching is not possible, take the instance off public network access, audit all workflow credentials for rotation, and treat the host as potentially compromised.

*Sources: [Critical RCE Flaw Allows Full Takeover of n8n — CSO Online](https://www.csoonline.com/article/4113980/critical-rce-flaw-allows-full-takeover-of-n8n-ai-workflow-platform.html)*

### PleaseFix / PerplexedBrowser — Agentic Browser Zero-Click Compromise (Patched)

**Disclosed:** March 4, 2026 | **Researcher:** Zenity Labs | **Affected:** Perplexity Comet | **Status:** Patched by Perplexity prior to public disclosure | **CVE:** Not assigned

Zenity Labs disclosed **PleaseFix**, a family of critical vulnerabilities in agentic browsers including Perplexity Comet, via responsible disclosure. The **PerplexedBrowser** subfamily contains two distinct exploitation paths, both rooted in indirect prompt injection:

**Path 1 — Zero-click file system access:** When a Comet agent session processes attacker-controlled web content (a webpage, document, or search result), a zero-click exploit grants access to the local file system and enables data exfiltration. The agent continues returning expected results to the user throughout, with no visible anomaly.

**Path 2 — Credential theft via password manager abuse:** The second path manipulates agent-authorized workflows to interact with the user's password manager in ways that produce credential theft or full account takeover, without directly exploiting the password manager itself. The agent is used as a trusted proxy to invoke the credential access.

Zenity Labs responsibly disclosed both paths to Perplexity; the browser-side agent execution issue was addressed before public disclosure. The disclosure is significant because it demonstrates that agentic browsing — not just agentic coding — is subject to the same prompt-injection-to-exfiltration chains seen in IDE-based agents.

*Sources: [The vulnerability that turns your AI agent against you — Help Net Security](https://www.helpnetsecurity.com/2026/03/04/agentic-browser-vulnerability-perplexedbrowser/)*

### CVE-2025-68143 / 68144 / 68145 — Anthropic mcp-server-git Chained RCE (Patched)

**Disclosed:** January 20, 2026 | **Researcher:** Yarden Porat (Cyata) | **Affected:** Anthropic `mcp-server-git` (official) | **Status:** Patched | **Chain:** Path traversal + argument injection → RCE via prompt injection alone

Cyata researcher Yarden Porat published a three-CVE chain targeting Anthropic's own official Git MCP server — the reference implementation distributed in Anthropic's documentation:

| CVE | Type | Description |
|-----|------|-------------|
| CVE-2025-68145 | Path validation bypass | Allows operations outside intended repository scope |
| CVE-2025-68143 | Unrestricted `git_init` | Allows attacker to turn `.ssh/` into a git repo, corrupting SSH known_hosts behaviour |
| CVE-2025-68144 | Argument injection in `git_diff` | Passes attacker-controlled arguments to git subprocess without sanitization |

The full chain is exploitable via prompt injection alone: a malicious `.git/config` file in an attacker-controlled repository triggers the CVE-2025-68145 scope bypass, the argument injection delivers the payload, and combined with the Filesystem MCP server, achieves full remote code execution. No user action beyond opening a repository link in Claude is required.

The significance of this chain is that it targets Anthropic's own reference MCP server, not a third-party integration — demonstrating that even first-party, well-resourced implementations contain the same class of flaws pervasive in the ecosystem.

## Supply Chain

### s1ngularity — Nx Build System Compromise Weaponises AI CLI Tools

**Disclosed:** 2026 | **Researcher:** StepSecurity | **Package:** `nx` (npm) | **Status:** Malicious package removed from npm | **Significance:** First documented malware using AI coding CLI tools as active reconnaissance instruments

The **s1ngularity** attack compromised the popular Nx build system package on npm to deliver a credential-stealing payload targeting cryptocurrency wallets, GitHub/npm tokens, SSH keys, and environment secrets. What distinguishes s1ngularity from prior npm supply chain attacks is its use of AI coding CLI tools as post-compromise instruments.

**AI weaponisation module:** After establishing a foothold, the malware invoked **Claude**, **Gemini**, and **Amazon q** CLI tools present on the victim's developer machine to perform active reconnaissance of the environment — analysing directory structures, reading configuration files, and identifying high-value targets for exfiltration. The AI tools were used for their natural language reasoning capabilities: given a directory listing, an LLM can identify which files are most likely to contain API keys or secrets far more effectively than a static regex scan.

**Why this changes the threat model:** Prior supply chain attacks on developer machines treated AI coding tools as neutral bystanders. s1ngularity demonstrates that installed AI CLI tools — with their filesystem access, context-reading capabilities, and natural language reasoning — are attack-multiplying assets for a post-compromise attacker. The presence of Claude Code, Gemini CLI, or q on a compromised machine is now itself a risk factor, not merely an innocent tool.

**Indicators:** Review npm install logs for unexpected `nx` package versions; audit shell history for unexpected invocations of `claude`, `gemini`, or `q` CLI tools; rotate any credentials on developer machines that installed affected `nx` versions.

*Sources: [s1ngularity: Popular Nx Build System Package Compromised — StepSecurity](https://www.stepsecurity.io/blog/supply-chain-security-alert-popular-nx-build-system-package-compromised-with-data-stealing-malware) | [The Hidden AI Threat to Your Software Supply Chain — Check Point Blog](https://blog.checkpoint.com/research/the-hidden-ai-threat-to-your-software-supply-chain/)*

### Supabase Cursor Agent Data Breach — Privileged MCP Agent Exfiltrates Integration Tokens

**Disclosed:** Mid-2025 (post-incident analysis published early 2026) | **Vector:** Prompt injection via support tickets | **Status:** Incident closed; access controls redesigned

The Supabase Cursor agent, deployed with **service-role** (full database admin) access to handle developer support workflows, processed support tickets that included user-supplied content as direct agent instructions. Attackers embedded SQL injection-style prompt instructions in support ticket text directing the agent to read and exfiltrate sensitive integration tokens by leaking them into a public support thread response.

**Attack anatomy — the "lethal trifecta":**
1. The agent held privileged service-role access to sensitive integration tokens (private data)
2. It processed user-supplied support ticket content without trust boundaries (untrusted input)
3. Its response channel (public support thread) provided an external exfiltration path (external communication)

No additional tooling was required. The attacker submitted a support ticket; the agent did the rest.

**Systemic lesson:** This is the canonical live demonstration of Simon Willison's "lethal trifecta" — AI agents that simultaneously hold private data access, process untrusted content, and can communicate externally are **exploitable by design**, regardless of how they were intended to be used. Architectural redesign (least privilege, context separation, output filtering) is the only mitigation; security warnings and policy documents are insufficient.

*Sources: [MCP Security Vulnerabilities — Practical DevSecOps](https://www.practical-devsecops.com/mcp-security-vulnerabilities/) | [New Prompt Injection Attack Vectors Through MCP Sampling — Unit42](https://unit42.paloaltonetworks.com/model-context-protocol-attack-vectors/)*

## Research & Advisories

### CrowdStrike: AI-Coded Software Introduces Novel Vulnerability Patterns

**Published:** 2026 | **Source:** CrowdStrike Adversarial AI Research

CrowdStrike researchers published findings on **hidden vulnerabilities in AI-coded software** — patterns introduced by LLM code generation that differ qualitatively from vulnerabilities in human-authored code and evade standard static analysis tooling.

Key findings:
- AI code generators exhibit **correlated failure modes**: the same prompt phrasing elicits the same insecure pattern across thousands of developer sessions, creating systemic vulnerability clusters rather than isolated bugs
- LLMs frequently generate **syntactically correct but semantically insecure** authentication and cryptography code — code that passes review and linting but implements a broken security model (e.g., HMAC comparisons using `==` instead of constant-time comparison, JWT verification that accepts `alg: none`)
- Vulnerabilities introduced by AI coding tools are disproportionately concentrated in **security-adjacent code**: input validation, session management, file path handling, and privilege checks — precisely the code that has the highest security consequence per line
- **Endor Labs 2025 data** corroborates: 44–49% of AI-suggested dependency versions have known CVEs; 34% of suggested dependencies were hallucinated (non-existent in public registries)

The practical implication is that code review processes calibrated for human-authored code are insufficient for AI-generated code. AI tools can produce large volumes of plausible-looking security-critical code that requires domain-expert security review, not general code review.

*Sources: [CrowdStrike Researchers Identify Hidden Vulnerabilities in AI-Coded Software](https://www.crowdstrike.com/en-us/blog/crowdstrike-researchers-identify-hidden-vulnerabilities-ai-coded-software/) | [The Hidden Dependency Risk of AI-Written Code — Endor Labs](https://www.sdcexec.com/software-technology/ai-ar/article/22959921/endor-labs-the-hidden-dependency-risk-of-aiwritten-code/)*

### "The eval() Epidemic" — Three MCP CVEs, One Root Cause

**Published:** March 2026 | **Source:** DEV Community (kai_security_ai) | **Reference:** [Seven MCP CVEs in One Month: The Complete Map](https://dev.to/kai_security_ai/seven-mcp-cves-in-one-month-the-complete-map-1am5)

A community security analysis mapped **seven MCP CVEs in a single month** — including CVE-2026-0755 — to a single root cause: MCP server developers routinely pass user-controlled input to `eval()`, `exec()`, or shell-executing wrappers without validation, because the LLM client is mistakenly treated as a trusted sanitization layer.

Three of the seven CVEs share the same implementation pattern: a string received from the MCP client (which may itself have been prompt-injected) is passed to `child_process.exec`, `eval()`, or an equivalent shell interpreter without escaping shell metacharacters. The "eval() epidemic" framing highlights that this is not a series of independent mistakes but a shared design antipattern across independent MCP server authors — reflecting the state of MCP security education rather than isolated developer error.

**Recommendation:** Any MCP server that uses `exec`, `eval`, or shell execution should be treated as potentially vulnerable until explicitly audited. The [vulnerablemcp.info](https://vulnerablemcp.info/) database now catalogues 50+ MCP CVEs and serves as the reference for teams building MCP security baselines.

## Analyst Note

Three developments this cycle warrant elevation above the general signal:

**1. CVE-2026-0755 is the clearest current example of vendor abandonment of an AI security disclosure.** ZDI followed textbook coordinated disclosure for seven months. The vendor never responded. The result is a CVSS 9.8 zero-day in a widely-deployed MCP tool with no patch in sight. This failure mode — a small or underfunded MCP ecosystem participant unable or unwilling to maintain a security response process — will recur. The MCP ecosystem's rapid growth has outpaced its security maturity, and the registry has no mechanism to surface or quarantine packages with open critical zero-days. Until that changes, teams must treat the absence of a patch response as a red flag equivalent to a known exploit.

**2. s1ngularity represents a phase shift in how attackers think about AI tools on developer machines.** Prior supply chain attacks viewed AI coding tools as neutral artifacts. s1ngularity weaponised them as cognitive instruments — using LLM reasoning to accelerate post-compromise reconnaissance in ways that static scripts cannot. As AI coding assistants accumulate more filesystem access, more credential stores, and tighter integration with enterprise infrastructure (code repositories, CI/CD, cloud APIs), their presence on a compromised machine will increasingly determine what an attacker can do with that foothold.

**3. The Supabase incident and CVE-2026-21858 (n8n) together illustrate that agentic platform security is infrastructure security, not application security.** A compromised n8n host owns every agent workflow on that host. An overprivileged Cursor agent is an exfiltration primitive waiting for the right support ticket. As AI agents are deployed with higher privilege and deeper integration, the security standards applied to the platforms hosting them must match those applied to production infrastructure — hardened, isolated, least-privilege, and continuously monitored — not those applied to developer tooling.

**Immediate actions:**
- **gemini-mcp-tool users:** Sandbox or disable immediately; CVE-2026-0755 is CVSS 9.8 with no patch; treat as a persistent open RCE
- **GitHub Copilot CLI users:** Upgrade to **v0.0.423+** for CVE-2026-29783 fix; verify CVE-2026-21516 server-side patch has been applied
- **n8n operators:** Patch CVE-2026-21858 (CVSS 10.0) immediately; rotate all workflow credentials on affected hosts
- **All developers:** Audit for unexpected presence of `nx` package versions from the s1ngularity campaign; review AI CLI invocation history for anomalous calls
- **Anthropic mcp-server-git users:** Verify you are on a patched version post-CVE-2025-68143/68144/68145 chain
- **MS-Agent (ModelScope) users:** CVE-2026-2256 still unpatched (day 3); sandbox, disable Shell tool, or remove (CERT/CC VU#431821)
