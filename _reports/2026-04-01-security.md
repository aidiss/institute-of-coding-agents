**2026-04-01**

## Incidents & Vulnerabilities

### IDEsaster: 30+ CVEs Across AI Coding Tools

A six-month research campaign catalogued over **30 vulnerabilities across 10+ AI coding products** — GitHub Copilot, Cursor, Claude Code, Windsurf, Kiro.dev, Zed.dev, Roo Code, JetBrains Junie, Cline, and Gemini CLI — yielding **24 assigned CVEs** and an AWS security advisory (AWS-2025-019). The "IDEsaster" attack chain is novel in targeting the **base IDE layer** rather than AI-specific components, meaning 100% of tested tools were affected. The chain flows: **Prompt Injection → Tool Abuse → Base IDE Feature Exploitation**.

### Zero-Click Agentic RCE (Research)

Researchers demonstrated a zero-click scenario in which a Google Docs file triggered an IDE agent to fetch attacker-controlled instructions from an MCP server, execute a Python payload, and harvest secrets — entirely without user interaction. No specific CVE assigned; disclosed as proof-of-concept.

---

### .env File / Secrets Exposure — All Major Tools

Claude Code, Cursor, Copilot, Windsurf, Aider, and Cline **automatically include `.env` files** in agent context, transmitting API keys, database credentials, and tokens to LLM providers. Cursor's agent mode can additionally **bypass `.cursorignore`** restrictions via shell commands. No single CVE; systemic design issue across the category.

## Supply Chain

### LiteLLM PyPI Compromise — CVE-2026-33634 (CVSS 9.4) — **Actively Exploited**

**Threat actor:** TeamPCP (linked to LAPSUS$ collaboration per Wiz)  
**Affected versions:** litellm 1.82.7 and 1.82.8  
**Window:** ~3 hours, 2026-03-24 10:39–16:00 UTC  
**Downloads at risk:** ~3.4 million/day; ~95 million/month

**Attack chain:**
1. TeamPCP rewrote Git tags in the `aquasecurity/trivy-action` GitHub Action (2026-03-19) to point to a malicious release carrying a credential harvester.
2. LiteLLM's CI/CD pulled Trivy from apt without version pinning; the action exfiltrated the `PYPI_PUBLISH` token.
3. Attackers published `litellm 1.82.7` then `1.82.8` to PyPI using the stolen token.
4. A `.pth` file (`litellm_init.pth`) auto-executed a payload on every Python process startup.
5. Version 1.82.8 escalated to run malicious code **on every Python invocation** system-wide.

**Payload:** credential harvester targeting SSH keys, AWS/GCP/Azure tokens, Kubernetes configs, `.env` API keys, and database passwords; plus a Kubernetes lateral-movement toolkit and persistent backdoor.

**Broader campaign:** Same actor previously compromised **Checkmarx KICS** (2026-03-23) and **Telnyx** (2026-03-27) using credentials obtained from earlier stages of the same pipeline.

**Remediation:**
- Run `pip show litellm`; if 1.82.7 or 1.82.8, remove immediately and purge caches.
- Check for `~/.config/sysmon/sysmon.py` and associated systemd unit.
- In Kubernetes: audit `kube-system` for `node-setup-*` pods.
- Enable **PyPI Trusted Publishers (OIDC)** to eliminate long-lived tokens from CI.
- Pin all CI action dependencies to commit SHAs.

Sources: [The Hacker News](https://thehackernews.com/2026/03/teampcp-backdoors-litellm-versions.html) · [Datadog Security Labs](https://securitylabs.datadoghq.com/articles/litellm-compromised-pypi-teampcp-supply-chain-campaign/) · [Endor Labs](https://www.endorlabs.com/learn/teampcp-isnt-done) · [Wiz](https://www.wiz.io/blog/threes-a-crowd-teampcp-trojanizes-litellm-in-continuation-of-campaign) · [Snyk](https://snyk.io/articles/poisoned-security-scanner-backdooring-litellm/) · [ReversingLabs](https://www.reversinglabs.com/blog/teampcp-supply-chain-attack-spreads)

### Slopsquatting — AI Hallucinated Package Names as Attack Surface

Research across **16 LLMs** and **756,000 generated code samples** found that ~20% included hallucinated (non-existent) package names. Open-source models (DeepSeek, WizardCoder) hallucinated at 21.7%; commercial models (GPT-4-class) at 5.2%. Critically, **43% of hallucinated names were reproduced consistently** across 10 re-runs of the same prompt, making them predictable targets.

Attackers register the hallucinated names on PyPI/npm ahead of time. When a developer runs `pip install` on an AI-suggested package, they install the malicious payload. The attack requires no phishing or social engineering — the AI does the redirecting.

Autonomous coding agents (Cursor, Claude Code, Copilot Workspace) that can `pip install` without a human review step are the highest-risk surface: a single hallucinated dependency can propagate into production at machine speed.

**Mitigations:** lockfiles and hash pinning; dependency scanning before install; lower model temperature to reduce hallucination rate; RAG-based grounding (reduces hallucinations by up to 85% at some code quality cost); manual verification of any AI-suggested package name.

Sources: [Bleeping Computer](https://www.bleepingcomputer.com/news/security/ai-hallucinated-code-dependencies-become-new-supply-chain-risk/) · [Security Boulevard](https://securityboulevard.com/2026/03/coding-agents-widen-your-supply-chain-attack-surface/) · [Infosecurity Magazine](https://www.infosecurity-magazine.com/news/ai-hallucinations-slopsquatting/) · [Thinkpol](https://thinkpol.ca/2026/03/28/slopsquatting-the-supply-chain-attack-vibe-coding-made/)

### MCP Ecosystem Supply Chain Risks

Analysis of 7,000+ public MCP servers found **36.7% vulnerable to SSRF**. A comprehensive MCP vulnerability database ([vulnerablemcp.info](https://vulnerablemcp.info/)) now tracks **50+ documented MCP vulnerabilities, 13 critical**, across 32 researchers.

"Tool poisoning" — where an MCP server silently modifies its tool definitions after approval — enables post-deployment rug-pull attacks with no re-consent required.

## Research & Advisories

### OWASP LLM Top 10 (March 2026 Update)
OWASP reclassified **prompt injection** as the highest-severity single vulnerability category for deployed LLMs, above data poisoning, model theft, and insecure output handling. Indirect prompt injection (embedded in documents, emails, web pages, database content) now accounts for **>80%** of documented enterprise attack attempts; direct injection is below 20%.

### PromptPwnd — CI/CD Pipeline Injection
A new attack class targets AI agents connected to GitHub Actions / GitLab CI/CD. Malicious prompt injections trick agents into invoking privileged built-in tools, leading to information leakage or code execution in pipeline contexts. No single CVE; classification ongoing.

### Prompt Injection → RCE Escalation Path (Trail of Bits)
Trail of Bits documented the generic path from prompt injection to RCE in agentic systems, noting that "zero-click" and "one-click" RCE scenarios are now reproducible across multiple product categories. The key enabler is LLM inability to distinguish instructions from data at the architecture level. Source: [Trail of Bits Blog](https://blog.trailofbits.com/2025/10/22/prompt-injection-to-rce-in-ai-agents/)

### Palo Alto Unit 42 — MCP Sampling Attack Vectors
Unit 42 documented new prompt injection attack vectors through MCP's sampling mechanism, where a malicious server influences the model's intermediate reasoning steps without triggering tool-call consent dialogs. Source: [Unit 42](https://unit42.paloaltonetworks.com/model-context-protocol-attack-vectors/)

### AI-Generated Defense Code (War on the Rocks, March 2026)
Analysis of AI-generated code entering defense and critical-infrastructure contexts warns that foundation model compromises — unlike traditional supply chain attacks — can inject vulnerabilities into every codebase across every developer using every tool built on a compromised model, simultaneously and continuously. Source: [War on the Rocks](https://warontherocks.com/2026/03/your-defense-code-is-already-ai-generated-now-what/)

- Claude Code: 4% → **63% developer adoption** by February 2026

The blast radius of any single vulnerability in this category is correspondingly large.

## Key Mitigations (Cross-Cutting)

1. **Sandbox agent execution** — isolate from host filesystem, network, and credentials.
2. **Pin CI/CD action dependencies** to commit SHAs; never use floating version tags.
3. **Enable PyPI Trusted Publishers (OIDC)** to eliminate long-lived publishing tokens.
4. **Least-privilege tool scoping** — restrict what MCP servers and agent tools can access.
5. **Verify all AI-suggested package names** before install; use lockfiles and hash verification.
6. **Audit `.claude/settings.json`, `.mcp.json`, and `.cursor/` configs** in any repository before opening it.
7. **Never store live credentials in `.env` files** within AI-agent working directories.
8. **Treat MCP server tool definitions as untrusted** after initial approval; re-verify on version change.
